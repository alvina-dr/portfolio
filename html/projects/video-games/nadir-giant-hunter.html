Nadir Giant Hunter

2024 Unity Gamepad Localization Remapping

For our end of the year project, we had the idea to make a sort of mix between Spiderman and Shadow of the Colossus. In other words, a really dynamic controller and huge creatures to take down quickly.

We had three months to make the game with a team of seven. We were two programmers, and while I was working full time on the project, the other dev could only be there half of the time as they were working on two games.

This game used physics much more than any other game I had done before and led to lots of pretty complex actions. The first thing to design was the swinging part.

Swinging

For this part, I first tried to look into what Spiderman actually does. One of the big questions I had, was how do we know which point to use to swing. After searching a bit on the internet I found this conference that at some point explains how the developers of Spiderman themselves got inspired by Energy Hook's way of doing things : Classic Game Design Postmortem: Swinging with Spider-Man (GDC).

Basically what they did is they trace a cone of raycast that detects all the possible swinging points. In Energy Hook, the cone expands the longer the input to swing is held, in order to catch more possible points. In our case we decided to go with something pretty similar, we trace a cone of raycast. If a raycast hits something, it saves the collided object and the point of collision in a list. Once all collisions have been saved, we then calculate the distance between each one and a defined point that we call "perfect swinging point" which the game designers can move around to influence how we want our average swing curve to feel like. We then select the collision point that's the closest to our perfect swinging point.

Since in our game the player would be able to grapple at moving objects like the ennemies, we needed to save not only the position but also the object that we wanted to connect with. That way we make our grapple point a children of the object we want to use to swing - and it will follow the movement of its parent.

Now, in our game we had two raycast cone like this, which means that calculation done every frame was huge. To help with performance, what we did was to divide this calculation on several frames instead of just one. The calculation of the swinging point would be less precise but we would get more FPS.

The thing is, if you only do one cone raycast then you miss lots of opportunities to grapple at stuff in the environment. For example, you might miss the ceiling right above your head since it's in the middle of the cone. We decided to do something similar to the conference linked before and to make the cone grow overtime. This means that the cone will start with a really small radius and while the player holds the input to swing, the cone will get bigger and bigger. That way we include many more possibilities.

Example of radius step

The cone raycast had parameters like following :
Angle precision : The number of raycast in a circle => for example we could do a 360 raycast circle if we want a lot of precision - one per degree - or only 90 if we want to have more performance.
Minimum Radius : The minimum size of the cone radius, at which the cone starts when the player tries to swing.
Maximum Radius : The maximum size of the cone radius, reached if the swing input is not released for some times.
Raycast Length : This parameter was not in the ConeRaycast script but in the swinging parameters (in the player data). Still, it was a really important variable for the cone behavior.
Radius Precision : The number of step between the Minimum Radius and the Maximum Radius. The higher it is, the more precise the collision detection will be, but it will also cost more performance.

In Nadir the player can interact with three types of "spots" when they are close enough. The ennemis' weak spot, the dash spot and the bumpers.

Nadir is a gamepad first game, with fully remappable inputs.

Since we wanted to participate to the Busan Indie Connect Festival, I set up the Unity Localization System on the project. That way the game was playable in English, French and Korean.
